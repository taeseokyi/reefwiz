# 블루투스 통신 안정화 (RF 순단 대응) 및 시뮬레이터 회귀 검증

측정(`bin/measure_kh_once.py`)은 PC ↔ AquaWiz 사이를 **HC-06 블루투스 SPP(가상 시리얼 COM 포트)** 로
통신한다. 이 링크가 간헐적으로 끊겨 측정이 실패하던 문제와, 그 대응(코드 4계층 + 예방 keepalive),
그리고 하드웨어 없이 검증하는 소켓 시뮬레이터 회귀 테스트를 정리한다.

> **COM 포트 번호는 수시로 바뀐다**(재페어링·동글 교체·윈도우 업데이트 재열거). 실제 포트는 소스에
> 하드코딩하지 않고 단일 설정 파일 `bt_config.json`에서 읽는다 — [system-setup.md의 BT 포트 설정](system-setup.md#bt-포트-설정--단일-설정-파일-bt_configjson) 참조. 아래 본문의 `COM9`는 최초 진단(2026-06) 당시 값이다.

> 한 줄 요약은 [system-setup.md](system-setup.md#블루투스-통신-안정화--시뮬레이터-검증)에, 사건 기록은
> [measurement-ledger.md](measurement-ledger.md)(6/23 05:00)에 있다.

## 1. 문제 — HC-06 SPP RF 순단

- **링크:** 최초(2026-06)에는 COM9 = PC 내장 **인텔 Dual Band Wireless-AC 8265** 의 블루투스 SPP. **2026-07-21부터는 내장 8265의 간헐 장애(콤보칩 버스트) 때문에 USB CSR 동글로 전환**(내장 BT는 "사용 안 함")했고, 실제 COM 번호는 `bt_config.json`이 관리한다.
- **증상:** RF 링크가 간헐(버스트)적으로 끊긴다. 펌웨어는 살아있고, **드롭은 대부분 "명령을 보내려는 순간 이미 끊겨 있음"**(통신 도중엔 잘 안 끊김). 재연결하면 금방 붙고 재페어링·COM 번호 변경은 없다.
- **대표 사건(6/23 05:00):** 자동 측정이 첫 `tank` 측정에서 `[START]` 출력 후 통신이 완전히 멎어 실패(`dkh.dat`에 `05 0.000` 에러 표식). 수동 재실행(05:56)은 정상(`dKH 8.427`).
- **원인:** RF 마진 부족(거리) 아님 — 바로 옆에서도 끊김 → **주변 2.4GHz 간섭**으로 추정. HC-06은 ~20초 무통신 시 링크가 끊기는 경향도 있음(유휴 타임아웃).

### 호스트(Windows)측 조치

| 조치 | 이유 |
|------|------|
| **Wi-Fi 어댑터·BT 네트워크 장치 "사용 안 함"** | 유선랜 사용 중. 같은 콤보칩의 Wi-Fi가 미연결이어도 백그라운드 스캔으로 2.4GHz를 점유해 BT를 방해 |
| **USB 선택적 절전 OFF** (`powercfg ... 48e6b7a6... 0`, AC/DC) | 내장 BT도 내부 USB 버스라, 포트 절전 서스펜드가 RF 드롭을 유발 |
| (참고) 내장 BT 라디오 전원관리 | 이미 "절전을 위해 끔" 비활성 상태 |

## 2. 대응 — `measure_kh_once.py`의 `send()` 4계층 + keepalive

모든 펌웨어 명령은 `send()`(모터는 `send_motor()`)를 거친다. RF 순단에 대해 다음을 보장한다.

1. **송신 전 연결 확인** — 모든 명령은 보내기 전 `ensure_link()`(부작용 없는 `status` 핑)로 링크 생존을 확인한다. (드롭이 대개 "보낼 때 이미 끊김"이므로 이 단계가 핵심.)
2. **끊겼으면 재연결** — `reconnect()`가 포트를 `close→open` 하여 HC-06 재연결을 유도하고, `status` 응답으로 복구를 확인한다.
3. **보낸 뒤 연결 문제면 재연결 후 재시도** — 송신/수신 예외 또는 응답 미수신(타임아웃)이면 재연결 후 같은 명령을 재시도한다.
4. **재시도 횟수 상한** — `SEND_RETRY_MAX`(기본 3)회까지. 소진하면 기존 백스톱(측정 `FAIL_MAX`·phase 상한·`_safe_cleanup`·에러 래치)이 처리한다.
5. **모터는 재시도 시 먼저 정지(`mNs`) 후 재송신** — 죽은 링크에 미전달됐든 진행 중이든, 정지 명령으로 멈춘 뒤 재송신해 **펌프 중복 구동(과주입/오버플로)을 방지**한다. (측정·계산·솔레노이드는 멱등/읽기전용이라 그냥 재송신해도 안전.)

**예방 — keepalive:** 측정 간격(30초)·모터 동작(60~85초) 등 유휴 구간 동안 `KEEPALIVE_SECS`(12초)마다 빈 줄(`\r\n`, 펌웨어가 조용히 무시)을 보내 링크를 깨워 둔다 → HC-06 ~20초 유휴 타임아웃 드롭을 애초에 줄인다. 그래도 끊기면 위 4계층이 복구한다.

`calkh`·`calref`(`--setref`)는 동일한 `send()`/측정 루프를 타므로 같은 대응이 자동 적용된다.

### 관련 상수 (실전값)

| 상수 | 기본값 | 의미 |
|------|--------|------|
| `RECONNECT_TRIES` | 5 | `reconnect()` 1회당 `close→open` 재연결 시도 횟수(진짜 순단은 1~2회면 붙음) |
| `RECONNECT_BACKOFF` | (1,1,2,2,3) | 재연결 시도별 대기(초). 범위 넘으면 마지막값 유지 |
| `SEND_RETRY_MAX` | 3 | `send()` 1회당 재시도 횟수(튜닝 가능) |
| `KEEPALIVE_SECS` | 12 | 유휴 keepalive 간격(초) — HC-06 ~20초 임계 아래 |
| `MEAS_READ_TIMEOUT` | 20 | 측정 1회 `[OK]` 대기 상한(초). 펌웨어 측정 ~8초이므로 20초면 RF 드롭 판정 |
| `LINK_PING_TIMEOUT` | 3 | `ensure_link`/`reconnect`의 `status` 핑 응답 대기 상한(초) |
| `write_timeout=5` | 5 | 시리얼 write 블로킹 상한(초) — 반열림 BT COM에서 `write()` 무한 블로킹(좀비) 방지 (§2.5) |
| `LINK_RETRY_INTERVAL` | 60 | 평탄 phase 중 링크 완전 사망 시 재접속 시도 간격(초) — phase 마감까지 반복 (§2.5) |

> 펌웨어는 `m1s`~`m4s` 개별 모터 정지를 지원한다(응답 `[Mn] 정지`).

### 2.5 장기 링크 사망 보강 (2026-07-03)

7/2 21시 자동 측정이 도저 설치 작업 중 BT가 장시간 끊기며 실패했다. 사후 분석 결과 단순 실패가
아니라 **좀비**였다: keepalive의 `ser.write()`가 반열림 COM 포트에서 **무한 블로킹**(당시
`write_timeout` 미설정) → 스케줄러 5h 상한(02:00)에 강제 종료 → 에러 행 기록·정리(`_safe_cleanup`)가
전부 미실행. 이에 4종을 보강했다(순단 대응 위에 얹힘, 시뮬 시나리오 [18][19]로 검증).

1. **`write_timeout=5`** — write 블로킹에 상한. `SerialTimeoutException`은 `SerialException`의
   하위라 기존 예외 처리(재연결 경로)가 그대로 받는다.
2. **평탄 phase 중 링크 사망 = 끈질긴 재접속** — `FAIL_MAX` 포기 대신 `_wait_link_recovery`가
   phase 마감(`PHASE_MAX_SECS`)까지 `LINK_RETRY_INTERVAL`(60초) 간격으로 재접속을 시도한다.
   폭기는 유지되므로 평형이 보존되고 모터는 정지 상태라 무해 — 복구되면 측정을 이어간다.
3. **호스트 구제** — 링크 사망으로 `calkh`를 못 불러도 phase 데이터(tank·ref pH)가 온전하면,
   시작 시 `status`에서 캐시해 둔 refKH·온도로 **호스트가 동일 차동식으로 dKH를 계산**해
   **음수 표식**으로 기록·발행한다(0.0 에러 래치 회피). 링크는 살았는데 장비 문제(KCl 소크 실패 등)면
   기존대로 0.0 래치를 유지한다.
4. **스케줄러 등록 파일 정합** — `register_measure_kh.ps1`의 `ExecutionTimeLimit`을 55분(V3
   잔재)에서 5h로 갱신(실제 등록은 이미 5h였고 파일만 낡아 있던 재등록 지뢰 제거).

7/3 13시 첫 실전 런에서 구제 캐시 라인 정상 출력·BT 이벤트 0건으로 확인 완료(링크 사망 경로
자체는 실전 미발생 — 시뮬레이터로만 검증됨).

### 배포 안전 — 테스트 타이밍 패치는 in-memory 한정

테스트는 `import measure_kh_once` 후 `mk.MEAS_INTERVAL=0.02`, `mk.FAIL_MAX=2`,
`mk.MEAS_READ_TIMEOUT=0.5`, `mk.LINK_PING_TIMEOUT=0.3` 등으로 **메모리상 모듈만** 패치해 빠르게
돈다. **소스 파일의 실전 상수는 바뀌지 않으므로**(§2 표) 배포본은 정상 동작한다. 이 분리를 위해
매직넘버를 모듈 상수로 빼 두었다(테스트에서 패치 가능하도록).
